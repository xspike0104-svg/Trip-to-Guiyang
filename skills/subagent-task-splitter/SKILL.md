# Subagent Task Splitter

将复杂任务分解为细粒度子agent并行执行的技能。

## 核心原则

1. **单一职责**：每个子agent只做一个明确的任务
2. **并行执行**：独立任务同时运行，减少总等待时间
3. **粒度适中**：任务太小会创建过多子agent，太大达不到效果
4. **避免重复**：各子agent任务不重叠
5. **递归拆分**：如果拆分的子agent仍然较大，继续拆分直到每个只做一小部分
6. **⚠️ 永远不要用完context！最多用到90%**

## 使用场景

- 复杂旅行规划（多地点、多主题研究）
- 批量信息搜索和收集
- 并行技能安装（ClawHub多个技能）
- 大型报告的多章节同时撰写
- 竞品分析、市场研究等多维度分析

## 工作流程

### Step 0: Context健康检查（每次必做）

**首先检查当前context使用率**
- 如果超过70%，先保存进度，启动新session处理
- 如果在70%以下，继续执行

### Step 1: 识别任务类型

判断任务是否需要分解：
- 任务是否涉及多个独立研究方向？
- 信息收集是否可并行？
- 是否有多个人员/地点/主题需要独立研究？
- 预计上下文是否会超出限制？

**需要分解的典型场景**：
```
❌ "研究贵州旅行全部信息" （太大）
✅ 分解为：景点研究、交通研究、住宿研究、美食研究

❌ "安装10个技能并测试" （太大）
✅ 分解为：每批2-3个技能分批安装
```

### Step 2: 任务分解

将主任务拆分为3-6个细粒度子任务：

**分解维度**：
- **按信息类型**：景点、交通、住宿、美食、装备
- **按地域**：成都、重庆、贵阳、荔波
- **按阶段**：调研、对比、规划、确认
- **按功能**：搜索、安装、配置、测试

**粒度标准**：
| 任务复杂度 | 子agent数量 | 每任务预计耗时 |
|-----------|------------|--------------|
| 简单（单主题）| 2-3个 | 1-2分钟 |
| 中等（多主题）| 3-5个 | 2-5分钟 |
| 复杂（大型项目）| 5-8个 | 5-10分钟 |

### Step 3: 创建子agent

使用 `sessions_spawn` 并行创建：

```python
# 并行启动多个子agent
sessions_spawn(
    mode="run",      # 或 "session" 如果需要持续对话
    runtime="subagent",
    task="具体的研究任务描述",
    timeoutSeconds=60-120  # 根据任务复杂度调整
)
```

**任务描述规范**：
- 明确具体的研究内容
- 指定使用的技能或工具
- 说明输出格式和期望结果
- 避免过于宽泛的描述

### Step 4: 递归检查（重要！）

**拆分后再次评估每个子agent任务**：
- 如果某个子agent任务仍然太大（涉及多个子主题）
- **立即继续拆分**该子agent为更细粒度的任务
- 例如：
  ```
  ❌ "研究贵阳所有景点" （可能太大）
  ✅ 拆分为：
      - 黔灵山公园信息
      - 甲秀楼信息  
      - 青岩古镇信息
  
  ❌ "安装并配置5个skills" （太大）
  ✅ 拆分为：
      - 安装 pdf-extract
      - 安装 ppt-generator
      - 配置 skills环境
  ```

### Step 5: 收集和整合结果

等待所有子agent完成后：
1. 汇总各子agent的研究结果
2. 识别信息 gaps
3. 如需要，补充启动新的子agent

### Step 6: 输出最终结果

向用户呈现整合后的完整报告或计划。

## 子agent任务设计模板

```
## [任务编号] [具体研究内容]

**研究目标**：[明确的研究目标]

**具体任务**：
1. [具体任务1]
2. [具体任务2]
3. [具体任务3]

**使用的技能/工具**：[指定技能]

**输出格式**：[期望的输出]
```

## ⚠️ Context使用率监控

**硬性规则：Context使用率永远不要超过90%**

### 如何监控
- 定期使用 `session_status` 查看当前context使用量
- 当使用率超过80%时：
  1. 立即停止当前任务
  2. 保存当前进度到文件
  3. 将剩余任务分配给新的子agent

### 预防措施
- 每次启动子agent前，评估剩余context是否足够
- 大任务在context达到70-80%时就应该拆分
- 使用 `sessions_spawn` 时设置合理的 `timeoutSeconds`

### 处理流程
```
检测到context > 80%
    ↓
保存当前进度到文件
    ↓
启动新子agent完成剩余任务
    ↓
整合结果返回用户
```

## 实用技巧

### 控制并发数量
- 建议同时运行的子agent不超过5个
- 如果任务更多，按批次执行（每批3-4个）

### 设置合理超时
- 简单搜索任务：60秒
- 中等研究任务：90秒
- 复杂多步任务：120秒

### 任务状态追踪
```
| 子Agent | 任务 | 状态 |
|---------|------|------|
| #1 | 景点研究 | ✅完成 |
| #2 | 交通研究 | ⏳进行中 |
| #3 | 住宿研究 | ✅完成 |
```

### 错误处理
- 如果子agent超时，可以：
  1. 重新启动该子agent
  2. 使用已有部分结果
  3. 自己补充该部分信息

**原始任务**："帮我规划贵州5天旅行"

**分解方案**：

| 子Agent | 任务内容 | 使用的技能/工具 |

```python
# 并行启动5个子agent
subagent #1: 研究黄果树瀑布
subagent #2: 研究贵阳景点

```

## 常见问题

**Q: 子agent数量多少合适？**
A: 通常3-6个效果最好。少于3个可能分解不够，多于8个管理成本高。

**Q: 如何避免结果重复？**
A: 在任务描述中明确各子agent的研究范围，使用不同的分解维度。

**Q: 子agent失败了怎么办？**
A: 可以重新运行失败的子agent，或者自己补充该部分信息。

**Q: 什么时候不应该分解？**
A: 简单明确的任务（如"查询天气"）、需要连贯上下文的任务（如"写一篇连贯的文章"）。
